```java
@GetMapping("/layout")  // ← 第一个定义
```

#### Java Spring 的 Mapping 规则

**关键规则**：当有多个方法使用相同的 `@GetMapping` 路径时：
1. **Java编译器** 允许重载（方法签名不同）
2. **Spring 框架** 在运行时只选择**最后定义的一个**
3. **结果**：第二个定义（512行）**覆盖**第一个定义（217行）

**证明**：
```java
// SeatingController.java

@GetMapping("/layout")  // 第217行：被定义
public Result<VenueLayoutResponse> getSeatLayout(@RequestParam Long venueId) {
    // 实现：buildVenueLayout(venueId)
}

// ... 其他代码 ...

@GetMapping("/layout")  // 第512行：后来又定义
public Result<Object> getSeatLayout(@RequestParam Long conferenceId) {
    // 占位符：return Result.ok("开发中");
}

// 结果：请求 /api/seating/layout?venueId=123
//      Spring 选择使用第二个方法
//      返回占位符：{ "code": "0", "data": "获取座位平面图成功（开发中）" }
//      前端无法解析座位数据
```

## 根本原因

### 直接原因

| # | 问题 | 位置 | 原因 |
|----|------|------|------|
| 1 | 后端API有两个 `@GetMapping("/layout")` | SeatingController.java | 代码合并/重构时重复定义 |
| 2 | 第二个是占位符 | 第512行 | 开发不完整，遗留的 TODO |
| 3 | Spring 使用后定义 | 运行时特性 | Java 的方法重载规则 |
| 4 | 前端调用返回占位符 | 第5225行 | 依赖了被覆盖的 API |

### 根本原因

#### 原因1：后端代码整合时的重复定义

**推测场景**：
```
开发阶段：
  分支1 开发者A → 实现 /api/seating/layout API（第217行）
  分支2 开发者B → 添加占位符 /api/seating/layout API（第512行）
  
代码合并：
  两个分支的代码都被合并到主分支
  → 结果：两个 @GetMapping("/layout") 共存
  
运行时：
  Spring 框架选择最后加载的方法（第512行的占位符）
  → API 返回开发中提示，而非真实数据
```

#### 原因2：前端硬编码日程数据

**推测场景**：
```
开发阶段：
  后端 API 开发不及时 → 前端需要独立开发
  → 使用硬编码的模拟数据进行开发和测试
  
现在：
  后端 API 已完成，但前端未更新
  → 前端仍使用硬编码日程数据
  → 导致日程与真实会议数据不同步
```

### 直接影响

#### 1️⃣ 座位数据不加载

**链路**：
```

  → fetch('/api/seating/layout?venueId=...')
  → 返回占位符：{ "code": "0", "data": "获取座位平面图成功（开发中）" }

  → 座位数据为空
  → 显示默认测试数据
```

**表现**：
- ❌ 看不到真实座位布局
- ❌ 无法进行座位分配
- ❌ 统计数据错误

#### 2️⃣ 日程数据硬编码

**链路**：
```

  → 直接使用硬编码的3个模拟日程
  → 不调用后端 API
  → 日程数据与真实会议数据不同步
```

**表现**：
- ❌ 日程不会因为后端数据变化而更新
- ❌ 无法添加/删除日程
- ❌ 日程选择功能形同虚设

### 间接影响

**表现**：

**问题**：

### 修复2：升级前端日程加载为 API 优先

**目标**：从硬编码改为调用真实 API

**核心改变**：
```javascript
// 修改前：硬编码
const loadSchedules = async () => {
    schedulesList.value = [
        { id: 'sched_001', name: '开幕式', ... },
        { id: 'sched_002', name: '主题演讲', ... },
        { id: 'sched_003', name: '闭幕式', ... }
    ];
}

// 修改后：API 优先 + 降级
const loadSchedules = async () => {
    if (context.conference?.id) {
        try {
            const response = await fetch(`/api/schedule/all?meetingId=${context.conference.id}`);
            if (response.ok) {
                const data = await response.json();
                schedulesList.value = data.data || [];
                return;  // 成功
            }
        } catch (e) {
            // API 不可用，继续到降级
        }
    }
    // 降级：使用本地模拟数据
    loadSchedulesFromLocal();
}
```

**降级方案**：
- **优先级1**：调用 `/api/schedule/all?meetingId={conferenceId}`
- **优先级2**：如果 API 不可用，使用本地模拟数据
- **保障**：确保系统总是有数据，即使 API 失败

**验证**：
- ✅ API 可用时使用真实数据
- ✅ API 不可用时自动降级
- ✅ 用户体验无中断

## 最佳实践

### 1️⃣ 避免 API 重复定义

**规则**：
```java
// ❌ 错误：同一路径两个方法
@GetMapping("/api/resource")
public Result<A> method1() { }

@GetMapping("/api/resource")  // 重复！
public Result<B> method2() { }

// ✅ 正确：使用不同的路径或参数
@GetMapping("/api/resource")
public Result<A> method1() { }

@GetMapping("/api/resource/{id}")  // 不同路径
public Result<A> method1ById(@PathVariable Long id) { }

@GetMapping("/api/resource/detail")  // 不同语义
public Result<B> getDetail() { }
```

**检查方式**：
```bash
# 在 IDE 中搜索
@GetMapping("/layout")  # 不应该出现多次

# 或使用 grep
grep -n '@GetMapping("/layout")' *.java  # 最多1个结果
```

### 2️⃣ 避免前端硬编码数据

**规则**：
```javascript
// ❌ 错误：生产环境中的硬编码
```
