## 🔧 诊断流程

### 第1步：定位问题代码

**问题日志**：`GET /api/seating/layout 404`

**定位过程**：
```
浏览器错误 
  → 查看 seating-mgr.html:5215
  → 发现调用 /api/seating/layout?venueId={id}
  → API 返回 404
  → 推断：后端 API 实现有问题
```

### 第2步：检查后端实现

**位置**：SeatingController.java

**发现**：
```bash
grep -n '@GetMapping("/layout")' SeatingController.java
# 输出：
217     @GetMapping("/layout")   ← 第一个定义（真实实现）
512     @GetMapping("/layout")   ← 第二个定义（占位符）

结论：API 定义重复！
```

### 第3步：分析为什么 404

**原因分析**：

Java Spring 框架的行为：
1. 扫描所有 Controller 方法
2. 当发现 `@GetMapping("/layout")` 时
3. 后定义的方法**覆盖**先定义的方法
4. 运行时使用第 512 行的占位符版本
5. 占位符只返回硬编码字符串
6. 前端无法解析座位数据
7. 座位列表为空

**链式反应**：
```
API 返回占位符
  ↓
前端解析失败
  ↓
座位数据为空
  ↓
参会人员列表为空
  ↓
调用 initializeDefaultAttendees()
  ↓
显示测试数据
```

### 第4步：检查前端日程加载

**位置**：seating-schedule-manager.js:65

**发现**：
```javascript
const loadSchedules = async () => {
    // 模拟从后端加载日程数据
    schedulesList.value = [
        { id: 'sched_001', name: '开幕式', ... },
        { id: 'sched_002', name: '主题演讲', ... },
        { id: 'sched_003', name: '闭幕式', ... }
    ];
}
```

**分析**：
- ❌ 没有调用任何 API
- ❌ 完全硬编码日程数据
- ❌ 函数名说"加载"，实际是"模拟"

**影响**：
- 日程显示为硬编码的 3 个
- 切换日程无效（数据相同）
- 无法与后端数据同步

## 🔨 修复方案

### 修复1：删除后端重复 API

**目标**：让真实实现被 Spring 使用

```diff
- 删除第 508-520 行：@GetMapping("/layout") - 占位符版本
+ 保留第 217 行：@GetMapping("/layout") - 真实实现
```

**效果**：
```
请求：GET /api/seating/layout?venueId=123
修复前：返回 { "data": "获取座位平面图成功（开发中）" }
修复后：返回 { "data": { "seats": [...], "stats": {...} } }
```

### 修复2：升级前端日程为 API 优先

**目标**：从硬编码改为调用真实 API

```diff
- const loadSchedules = () => {
-     schedulesList.value = [硬编码日程];
- }

+ const loadSchedules = async () => {
+     // 优先级1：API 加载
+     const response = await fetch('/api/schedule/all');
+     if (response.ok) {
+         schedulesList.value = (await response.json()).data;
+         return;
+     }
+     // 优先级2：本地降级
+     loadSchedulesFromLocal();
+ }
```

**效果**：
- ✅ 日程与后端同步
- ✅ API 可用时使用真实数据
- ✅ API 不可用时自动降级
- ✅ 零用户体验中断

## 🎓 学到的经验

### 经验1：API 不能重复定义

**规则**：同一路径的 `@GetMapping` 只能定义一次

**防预**：
- 代码审查时检查重复定义
- 构建时添加检查脚本
- IDE 配置重复检测

### 经验2：前端不能硬编码生产数据

**规则**：生产环境的数据必须来自 API

**防预**：
- 区分测试数据和生产数据
- 测试数据用 `// TEST_DATA` 标记
- 代码审查时检查硬编码数据

### 经验3：实现降级方案

**规则**：优先使用 API，API 不可用时有备选方案

**模式**：
```
优先级1：API 可用 → 返回真实数据
优先级2：API 不可用 → 使用备选方案
结果：用户总是有数据
```
