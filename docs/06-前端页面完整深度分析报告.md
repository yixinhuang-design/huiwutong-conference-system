## 一、项目架构总览

### 1.1 技术栈

| 层面 | 技术 |
|------|------|
| 框架 | Vue 3.3.4 (通过 CDN 全局引入，非 SFC 模式) |
| UI | 自定义 CSS (glassmorphism 玻璃拟态风格), Font Awesome 6.x |
| HTTP | 原生 `fetch` API (无axios) |

## 六、seating-mgr.html (11459行) — 智能排座 ⚠️最大最复杂页面

### 6.1 Vue组件结构 (Vue 3 Composition API)

- **使用 setup() 函数**: 这是唯一使用Composition API的页面
- **ref()**: conference, seatingAreas, attendees, currentScheduleId, loading, 等大量响应式变量
- **计算方法**: updateStatistics(), createSeatingArea(), 等
- **引入模块**: seating-optimization-patch.js, conference-context.js, xlsx

### 6.2 API调用

| 方法 | URL | HTTP方法 | 端口 | 备注 |
|------|-----|---------|------|------|
| loadSeatingData L5227 | `http://localhost:8086/api/seating/layout/load?conferenceId=&scheduleId=` | GET | 8086 | 加载座位布局 |
| loadSeatingData L5357 | `http://localhost:8082/api/registration/list?conferenceId=&page=1&pageSize=1000` | GET | 8082 | 加载已通过报名人员 |
| saveLayoutToBackend L10713 | `http://localhost:8086/api/seating/layout/save` | POST | 8086 | 保存布局到数据库 |

### 6.3 数据流

1. 先从8086加载座位布局 → 从layoutJson恢复完整区域
2. 再从8082加载已通过报名人员(status=1) → 映射为参会者列表
3. 保存时将区域+座位+参会者序列化POST到8086

### 6.4 潜在缺陷

1. **🔴 P0**: 11459行的巨型单文件，极难维护
2. **🟡 P1**: 参会人员一次性加载pageSize=1000，大型会议可能不够
3. **🟡 P1**: API失败时 fallback 到 localStorage，可能造成数据不一致
4. **🟡 P1**: saveLayoutToBackend 失败时只是 console.warn，用户无明显提示
5. **🟡 P1**: Vue本地文件 `libs/vue/vue.global.prod.js` 而非CDN (其他页面用CDN)，版本可能不一致
6. **🟢 P2**: initializeDefaultAttendees() 在无真实数据时创建测试数据，可能被误认为真实数据

## 二十五、关键架构风险

1. **无前端路由框架**: 每个功能一个HTML文件，无法共享状态，每次跳转都重新加载Vue和所有资源
2. **API端口硬编码**: 所有 `localhost:80xx` 直写在代码中，部署到生产环境需要全局替换
3. **无网关统一代理**: 前端直连8个后端微服务端口，CORS/安全/负载均衡全靠各服务自行处理
4. **无全局状态管理**: 页面间通过 localStorage + URL参数传递数据，容易丢失或不同步
5. **CDN依赖**: Vue/FontAwesome/ECharts依赖外网CDN，离线环境无法使用 (seating-mgr.html用了本地libs但其他页面没有)
6. **代码重复**: 多个页面有相同的会议上下文初始化逻辑、相同的UI工具方法、相同的认证检查
7. **N+1 HTTP请求模式**: task-management.html中loadStaffAndMembers和loadTasksFromAPI均存在循环逐个请求问题(v2.0新增)
8. **第三方API Key泄露风险**: 高德地图API Key明文写在HTML的script标签src中，部署后任何人可直接提取使用(v2.0新增)
9. **X-Tenant-Id全局硬编码**: 至少在index/business-data/system-monitor/archive等6+个页面中硬编码同一租户ID(v2.0新增)
