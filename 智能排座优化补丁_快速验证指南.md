## 🔍 故障排查

### 问题1：浏览器控制台仍然显示"参会人员为空"

**原因可能**：
1. 浏览器缓存未清除
2. 后端未重启
3. 数据库中没有座位数据

**解决方案**：
```bash
# 方案A：清除浏览器缓存（最有可能）
Ctrl + Shift + Delete → 清除所有

# 方案B：强制刷新
Ctrl + F5

# 方案C：检查后端是否已重启
curl http://localhost:8080/api/seating/venues/2030309010523144194
```

### 问题2：API仍然返回404

**原因可能**：
1. SeatingController.java 未重新编译
2. 后端服务用旧版本运行
3. 代理或防火墙阻止

**解决方案**：
```bash
# 检查后端进程
ps aux | grep java

# 杀死旧进程
kill -9 <pid>

# 重新编译和启动
mvn clean package
./target/conference-backend-1.0.0.jar
```

### 问题3：日程列表为空或仍是模拟数据

**原因可能**：
1. 数据库中没有日程数据
2. 会议ID不正确
3. API端点变更

**解决方案**：
```bash
# 检查会议ID
# URL中: ?conferenceId=2030309010523144194

# 查看API响应
curl "http://localhost:8080/api/schedule/all?meetingId=2030309010523144194"

# 如果返回空数组，需要在数据库中添加日程数据
```

## 🎓 工作原理

### 修复前的问题链路
```
seating-mgr.html 加载
  ↓
loadSeatingData() 调用 /api/seating/layout?venueId=123
  ↓
SeatingController 中有两个 @GetMapping("/layout")
  ↓
Java 使用后定义（第512行的占位符版本）
  ↓
返回: { "code": "0", "data": "获取座位平面图成功（开发中）" }
  ↓
前端解析失败，座位数据为空
  ↓
显示："参会人员为空，补充默认测试数据"
```

### 修复后的数据流
```
seating-mgr.html 加载
  ↓
scheduleManager.loadSchedules() 
  ├─ 调用 /api/schedule/all?meetingId=123
  └─ 返回真实日程列表 ✅
  
loadSeatingData() 调用 /api/seating/layout?venueId=123
  ↓
SeatingController 中只有一个 @GetMapping("/layout")（第217行）
  ↓
调用 buildVenueLayout(venueId)
  ↓
返回完整的座位平面图数据 ✅
  ↓
前端正确加载座位，显示真实数据 ✅
```
