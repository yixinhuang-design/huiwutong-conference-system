## 问题根因分析

### 可能原因1：租户ID不匹配

**假设**：
- 前端发送：`X-Tenant-Id: 2027317834622709762`（默认值）
- 后端currentTenantId()返回：可能是不同的值
- 数据被存到了不同租户ID下

**验证SQL**：
```sql
SELECT * FROM conf_registration 
WHERE meeting_id = 2030309010523144200;
-- 如果无结果，尝试：
SELECT DISTINCT tenant_id FROM conf_registration;
-- 如果有不同的tenant_id，说明问题出在租户ID
```

**如何确认**：
- 检查X-Tenant-Id实际被设置的值
- 检查currentTenantId()的实现

---

### 可能原因2：会议ID被修改或截断

**假设**：
- 前端发送：`conferenceId: 2030309010523144200` (Long)
- 后端某个环节进行了转换
- 数据库中实际存的是不同值

**验证SQL**：
```sql
SELECT DISTINCT meeting_id FROM conf_registration;
-- 检查是否有类似但不完全相同的ID，如：
-- 203030901052314420
-- 30309010523144200
-- 等等
```

---

### 可能原因3：查询字段错误

**假设**：
- 用户查询时使用了错误的列名
- 或者数据根本没有被插入

**验证SQL**：
```sql
-- 检查表结构
DESC conf_registration;
-- 应该看到 meeting_id BIGINT 列

-- 检查是否有任何报名记录
SELECT COUNT(*) FROM conf_registration WHERE deleted = 0;
-- 如果count=0，说明根本没插入成功
```

---

### 可能原因4：租户隔离或多数据库

**假设**：
- 系统使用了多租户架构
- 不同租户的数据存在不同数据库或schema
- 用户查询时使用的database不正确

**验证SQL**：
```sql
-- 检查当前database
SELECT DATABASE();

-- 列出所有databases
SHOW DATABASES LIKE 'conference%';

-- 可能的database有：
-- conference_registration
-- conference_auth
-- conference_meeting
-- conference_seating
-- 等等
```

---

### 可能原因5：提交验证失败但返回成功

**假设**：
- 后端validateDynamicFields验证失败
- 但异常被捕获且隐蔽处理
- 返回了虚假的成功

**验证**：检查后端日志中是否有错误
```
[ERROR] 动态字段必填: ...
[ERROR] 字段校验失败: ...
```

---

### 可能原因6：transaction回滚

**假设**：
- 后端成功处理并返回了Registration对象
- 但由于某个后续操作失败，transaction被回滚
- 数据实际上没有保存

**验证**：检查后端日志中是否有回滚信息

### 方案C：添加数据一致性检查

在insert后，立即re-query确认数据已保存：
```java
Registration saved = registrationMapper.selectById(registration.getId());
if (saved == null) {
    logger.error("数据插入后查询失败！");
    throw new BusinessException("报名保存失败");
}
if (!Objects.equals(saved.getConferenceId(), request.getConferenceId())) {
    logger.error("保存的conferenceId不匹配！期望:{}, 实际:{}", 
        request.getConferenceId(), saved.getConferenceId());
    throw new BusinessException("报名数据保存错误");
}
```
