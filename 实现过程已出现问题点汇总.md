## 问题7：租户ID获取失败 - TenantContextHolder返回null

**问题表现：**
- 会议服务API返回401错误：`未获取到租户信息，请检查认证信息`
- IDEA控制台无任何 `[TenantFilter]` 日志输出
- Token有效但后端无法提取租户ID

**问题原因（根本原因）：**
- `TenantFilter` 位于 `com.conference.common.tenant` 包
- `ConferenceMeetingApplication` 使用 `@SpringBootApplication` 注解
- Spring Boot默认只扫描主类所在包：`com.conference.meeting.*`
- `TenantFilter` 未被Spring容器扫描注册，Filter从未执行
- 导致 `TenantContextHolder.setTenantId()` 从未被调用
- `MeetingController` 中 `TenantContextHolder.getTenantId()` 返回null

**问题分析链路：**
```
请求 → Servlet容器
       ↓ (TenantFilter未注册，跳过)
       ↓
    MeetingController.createMeeting()
       ↓
    TenantContextHolder.getTenantId() → 返回 null
       ↓
    检查 tenantId == null → true
       ↓
    返回 401 Unauthorized
```

**修复方案：**
在 `ConferenceMeetingApplication` 类上添加 `@ComponentScan` 注解：
```java
@SpringBootApplication
@ComponentScan(basePackages = {
    "com.conference.meeting",
    "com.conference.common"
})
@EnableDiscoveryClient
@MapperScan("com.conference.meeting.mapper")
public class ConferenceMeetingApplication {
```

**修复后链路：**
```
请求 → Servlet容器
       ↓
    TenantFilter.doFilter()
       ↓ 从JWT Token提取租户ID
       ↓
    TenantContextHolder.setTenantId(2027317834622709762)
       ↓
    MeetingController.createMeeting()
       ↓
    TenantContextHolder.getTenantId() → 返回 2027317834622709762
       ↓
    创建会议成功，返回201
```

**文件位置：** 
- `backend/conference-backend/conference-meeting/src/main/java/com/conference/meeting/ConferenceMeetingApplication.java`

**影响范围：** 
- **阻塞所有会议服务API** - 所有需要租户隔离的接口都会返回401
- 认证服务不受影响（有独立的扫描配置）

**验证方法：**
1. 在IDEA中运行会议服务
2. 观察Console输出是否包含：`[TenantFilter] 处理请求...`
3. 提交创建会议后查看是否输出：`[TenantFilter] 从Token提取userId: X, tenantId: Y`

**相关代码位置：**
- Filter定义：`conference-common/src/main/java/com/conference/common/tenant/TenantFilter.java`
- Context持有者：`conference-common/src/main/java/com/conference/common/tenant/TenantContextHolder.java`

## 问题8：Bean定义冲突 - mybatisPlusInterceptor重复注册

**问题表现：**
- 启动会议服务时报错：`BeanDefinitionOverrideException`
- 错误信息：`Cannot register bean definition... 'mybatisPlusInterceptor'... since there is already... bound`
- 服务无法启动

**问题原因（连锁反应）：**
- 添加 `@ComponentScan` 扫描 `com.conference.common` 包后
- Spring容器扫描到 `com.conference.common.config.MybatisPlusConfig`
- 该配置类定义了 `@Bean mybatisPlusInterceptor()`
- 会议服务的 `ConferenceMeetingApplication` 中也定义了同名Bean
- Spring Boot默认 `spring.main.allow-bean-definition-overriding=false`
- 两个同名Bean冲突导致启动失败

**完整错误日志：**
```
org.springframework.beans.factory.support.BeanDefinitionOverrideException: 
Invalid bean definition with name 'mybatisPlusInterceptor' defined in 
com.conference.meeting.ConferenceMeetingApplication: Cannot register bean 
definition... for bean 'mybatisPlusInterceptor' since there is already 
[...defined in class path resource [com/conference/common/config/MybatisPlusConfig.class]] bound.

Description:
The bean 'mybatisPlusInterceptor', defined in com.conference.meeting.ConferenceMeetingApplication, 
could not be registered. A bean with that name has already been defined in class path resource 
[com/conference/common/config/MybatisPlusConfig.class] and overriding is disabled.

Action:
Consider renaming one of the beans or enabling overriding by setting 
spring.main.allow-bean-definition-overriding=true
```

**修复方案：**
删除会议服务中重复的Bean定义，使用common包中统一的配置：
```java
// 删除以下代码块
@Bean
public MybatisPlusInterceptor mybatisPlusInterceptor() {
    MybatisPlusInterceptor interceptor = new MybatisPlusInterceptor();
    interceptor.addInnerInterceptor(new PaginationInnerInterceptor(DbType.MYSQL));
    return interceptor;
}
```

**修复后的ConferenceMeetingApplication.java：**
```java
@SpringBootApplication
@ComponentScan(basePackages = {
    "com.conference.meeting",
    "com.conference.common"
})
@EnableDiscoveryClient
@MapperScan("com.conference.meeting.mapper")
public class ConferenceMeetingApplication {
    public static void main(String[] args) {
        SpringApplication.run(ConferenceMeetingApplication.class, args);
    }
    // mybatisPlusInterceptor() 方法已删除
}
```

**文件位置：** 
- `backend/conference-backend/conference-meeting/src/main/java/com/conference/meeting/ConferenceMeetingApplication.java`

**影响范围：** 
- 阻塞会议服务启动
- 问题7修复后的连锁反应

**根本原因分析：**
1. 问题7修复时添加了 `@ComponentScan("com.conference.common")`
2. 扫描到common包中的所有配置类
3. Common包中存在全局MyBatis-Plus配置
4. 未检查会议服务本地是否有重复配置
5. 两个模块独立开发时可能存在配置重复

**经验教训：**
- 添加ComponentScan前需审查common包中的所有Bean定义
- 避免在业务模块中重复定义框架级配置
- 应统一在common模块管理基础设施配置
- 业务模块只需关注业务逻辑，不应定义框架Bean

**验证方法：**
1. 在IDEA中重新运行会议服务
2. 观察启动日志是否正常加载TenantFilter
3. 服务成功启动不再报Bean冲突错误

## 问题9：缺少 Jakarta Validation 依赖

**问题表现：**
- 启动会议服务时报错：`NoClassDefFoundError: jakarta/validation/ConstraintViolationException`
- 错误信息：`Error creating bean with name 'globalExceptionHandler': Lookup method resolution failed`
- 服务无法启动

**完整错误日志：**
```
org.springframework.beans.factory.BeanCreationException: Error creating bean with name 'globalExceptionHandler': Lookup method resolution failed
Caused by: java.lang.IllegalStateException: Failed to introspect Class [com.conference.common.exception.GlobalExceptionHandler] from ClassLoader
Caused by: java.lang.NoClassDefFoundError: jakarta/validation/ConstraintViolationException
Caused by: java.lang.ClassNotFoundException: jakarta.validation.ConstraintViolationException
```

**问题原因（依赖传递问题）：**
1. `GlobalExceptionHandler` 位于 `conference-common` 模块
2. 该类处理 `ConstraintViolationException` 异常（参数校验异常）
3. common 模块虽然声明了 `spring-boot-starter-validation` 依赖
4. 但 meeting 模块未显式声明该依赖
5. 在某些情况下，传递依赖可能未正确加载到 classpath
6. 导致运行时找不到 jakarta.validation 相关类

**根本原因：**
- 添加 `@ComponentScan` 后，Spring容器加载了 `GlobalExceptionHandler`
- 该类在类加载时需要反射检查所有方法的参数和返回值类型
- 方法中引用了 `ConstraintViolationException`，但该类不在 classpath 中
- 类加载器抛出 `NoClassDefFoundError`

**修复方案：**
在 `conference-meeting/pom.xml` 中显式添加 validation 依赖：

```xml
<!-- Spring Boot Validation -->
<dependency>
    <groupId>org.springframework.boot</groupId>
    <artifactId>spring-boot-starter-validation</artifactId>
</dependency>
```

**修复后验证：**
```bash
cd backend/conference-backend
mvn package -pl conference-meeting -DskipTests
# 启动服务，应该不再报NoClassDefFoundError
```

**修改文件：**
- `conference-meeting/pom.xml`: 添加 spring-boot-starter-validation 依赖

**经验教训：**
- **多模块项目中，依赖传递不总是可靠的**
- **如果业务模块直接使用了common模块中引用的第三方类，应该显式声明依赖**
- **@ComponentScan扫描common模块后，common的所有依赖都可能被加载**
- **遵循"显式优于隐式"原则，避免依赖传递带来的不确定性**
- **添加ComponentScan后，要确保所有扫描到的类的依赖都可用**

## 问题10：JWT运行时类缺失与CORS报错连锁问题

**问题表现：**
- 前端控制台报错：`No 'Access-Control-Allow-Origin' header`，并出现 `net::ERR_FAILED 500`
- 会议服务日志显示 `TenantFilter` 已执行到Token校验阶段，但请求最终500
- 后端根因异常：`java.lang.ClassNotFoundException: cn.hutool.jwt.signers.JWTSignerUtil`

**关键日志：**
```
[TenantFilter] 处理请求: POST /api/meeting/create
[TenantFilter] 开始验证Token...
Servlet.service() ... Filter execution threw an exception
java.lang.ClassNotFoundException: cn.hutool.jwt.signers.JWTSignerUtil
   at com.conference.common.util.JwtUtils.validateToken(JwtUtils.java:106)
   at com.conference.common.tenant.TenantFilter.doFilter(TenantFilter.java:47)
```

**问题原因（根因）：**
1. `conference-meeting/pom.xml` 中内部依赖坐标写成了 `com.huiwutong:conference-common:1.0.0`
2. 实际多模块工程应使用 `com.conference:conference-common:1.0.0`
3. 导致运行时加载了错误来源/旧版本的common包，Hutool JWT相关类未正确进入运行时classpath
4. `TenantFilter` 调用 `JwtUtils.validateToken()` 时触发 `ClassNotFoundException`
5. 因为请求在过滤器阶段异常中断，响应头未正常补全，浏览器侧表现为CORS错误

**修复方案：**
1. 修正meeting模块的common依赖坐标：

```xml
<dependency>
   <groupId>com.conference</groupId>
   <artifactId>conference-common</artifactId>
   <version>1.0.0</version>
</dependency>
```

2. 在meeting模块补充CORS配置，确保跨域预检与业务请求都返回正确响应头：
- 新增 `conference-meeting/src/main/java/com/conference/meeting/config/CorsConfig.java`
- 配置 `CorsFilter`，放行 `Origin/Method/Header`

3. 重新按多模块构建：
```bash
mvn clean package -DskipTests -pl "conference-common,conference-meeting" -am
```

**修复后结果：**
- 会议服务可正常启动
- `TenantFilter` 可进入Token提取与验证流程
- 移除了 `JWTSignerUtil` 类缺失导致的500根因

**经验教训：**
- **多模块项目中，内部依赖坐标（groupId/artifactId）必须与父工程严格一致**
- **浏览器出现CORS报错时，要同步检查服务端是否先发生500异常**
- **Filter链中的ClassNotFoundException会被前端误判为跨域问题**

## 问题11：创建会议500 - meeting_code为空导致数据库约束失败

**问题表现：**
- 前端提交创建会议返回 `500 Internal Server Error`
- 后端日志中 `TenantFilter` 已完成Token校验并成功设置租户上下文
- SQL插入阶段失败，报错字段为 `meeting_code`

**关键日志：**
```
==>  Preparing: INSERT INTO conf_meeting ( id, tenant_id, theme, description, status, create_time ) VALUES ( ?, ?, ?, ?, ?, ? )
### Cause: java.sql.SQLException: Field 'meeting_code' doesn't have a default value
```

**问题原因：**
1. 前端提交的是 camelCase 字段（如 `meetingCode`, `meetingName`）
2. `MeetingCreateRequest` 使用了 `@JsonProperty("meeting_code")` 等 snake_case 映射
3. 导致反序列化后 `meetingCode` 等关键字段为空
4. `BeanUtils.copyProperties(request, meeting)` 后 `meeting.meetingCode = null`
5. 插入数据库时触发非空约束异常，返回500

**修复方案：**
1. 将 `MeetingCreateRequest` 改为 `@JsonAlias("snake_case")`，同时兼容 camelCase 与 snake_case 入参
2. 修正字段类型以匹配实体与前端：
   - `meetingType`: `Integer` → `String`
   - `locationLatitude/locationLongitude`: `BigDecimal` → `Double`
3. 在 `MeetingServiceImpl.createMeeting()` 增加基础参数校验：
   - 会议编码不能为空
   - 会议名称不能为空
   - 对非法请求返回业务错误（400），避免落到数据库500

**修改文件：**
- `backend/conference-backend/conference-meeting/src/main/java/com/conference/meeting/dto/MeetingCreateRequest.java`
- `backend/conference-backend/conference-meeting/src/main/java/com/conference/meeting/service/impl/MeetingServiceImpl.java`

**验证结果：**
- 会议服务编译通过（BUILD SUCCESS）
- Token校验链路正常（`TenantFilter` 已验证通过）
- 消除了 `meeting_code` 为空导致的数据库异常根因

## 问题12：创建会议500 - cover_image_url字段超长

**问题表现：**
- 前端创建会议返回 `500 Internal Server Error`
- 后端报错：`Data too long for column 'cover_image_url' at row 1`
- 日志显示 `cover_image_url` 参数为超长 `data:image/...;base64,...` 字符串

**关键日志：**
```
### Cause: com.mysql.cj.jdbc.exceptions.MysqlDataTruncation:
Data truncation: Data too long for column 'cover_image_url' at row 1
```

**问题原因：**
1. 前端将内联Base64图片（data URL）直接放入 `coverImageUrl`
2. 数据库 `conf_meeting.cover_image_url` 字段长度有限（通常VARCHAR）
3. Base64字符串远超字段上限，插入时触发数据截断异常

**修复方案：**
1. 在会议服务新增封面上传接口：`POST /api/meeting/upload/cover`（`MultipartFile`）
2. 服务端将图片保存到本地目录：`{user.dir}/uploads/meeting/cover/`，生成唯一文件名
3. 返回可访问URL（如：`http://localhost:8084/uploads/meeting/cover/{fileName}`）
4. 新增静态资源映射：`/uploads/** -> file:{user.dir}/uploads/`
5. 前端在选择封面时先上传图片，`form.coverImage` 保存为URL，再随创建会议请求提交
6. 保留后端兜底：若收到base64或超长URL，清空字段并记录告警，避免再次触发500

**修改文件：**
- `admin-pc/conference-admin-pc/pages/conference-create.html`
- `backend/conference-backend/conference-meeting/src/main/java/com/conference/meeting/controller/MeetingController.java`
- `backend/conference-backend/conference-meeting/src/main/java/com/conference/meeting/config/StaticResourceConfig.java`
- `backend/conference-backend/conference-meeting/src/main/java/com/conference/meeting/service/impl/MeetingServiceImpl.java`

**验证结果：**
- 会议服务编译通过（BUILD SUCCESS）
- 消除 `cover_image_url` 过长导致的数据库500根因

## 技术债务记录

### 1. JWT库冲突问题（未修复）

**问题描述：**
- 项目同时依赖两个JWT库：
  - `io.jsonwebtoken` (JJWT)
  - `cn.hutool.jwt` (Hutool)
- 存在两个不同的 `JwtUtils` 实现：
  - `com.conference.common.utils.JwtUtils` (静态方法，使用JJWT)
  - `com.conference.common.util.JwtUtils` (Spring Bean，使用Hutool)

**当前状态：**
- 登录服务和会议服务实际使用Hutool的HS384算法
- Token能够正常生成和验证
- 暂时未影响功能

**建议：**

## 经验总结

### 1. Spring Boot组件扫描规则
- `@SpringBootApplication` 默认只扫描主类所在包及子包
- 跨模块依赖的 `@Component` 需要显式配置 `@ComponentScan`
- Filter类必须被Spring容器管理才能注册到Servlet容器
- **添加ComponentScan前必须审查目标包中的所有Bean定义**
- **避免Bean名称冲突，尤其是框架级配置Bean**

### 2. 多模块配置管理
- **基础设施配置（如MyBatis-Plus）应统一在common模块**
- 业务模块不应重复定义框架级Bean
- ComponentScan会扫描所有`@Configuration`类
- 需要建立模块配置职责划分原则

### 3. 多模块依赖管理
- **多模块项目中，依赖传递不总是可靠**
- **业务模块直接使用common模块的第三方类时，应显式声明依赖**
- **ComponentScan扫描common后，common的所有类依赖都需要可用**
- **遵循"显式优于隐式"原则，确保依赖明确性**
- **内部模块依赖坐标必须和父工程保持一致（特别是groupId）**
- 启动失败时优先检查 `NoClassDefFoundError` 和 `ClassNotFoundException`

### 4. 前端资源降级策略
- 外部CDN应提供fallback方案
- 必填功能（如地址输入）不应依赖第三方服务
- 优先使用inline资源（如SVG头像）避免404

### 5. 跨域与后端异常联调
- 浏览器报CORS时先排查后端是否返回500
- 预检请求（OPTIONS）通过不代表业务请求（POST）一定成功
- 过滤器链异常会导致响应头不完整，从而放大为前端CORS表象

### 6. 认证Token管理
- 前后端localStorage key必须统一命名
- Token验证失败时应提供明确的错误提示
- 跨服务调用必须正确传递认证上下文

### 7. 调试技巧
- 后端无日志输出时优先检查Bean是否被注册
- 401错误优先检查认证链路：Token → Filter → Context → Controller
- 使用浏览器Network面板查看实际请求头和响应体
- **Bean冲突错误仔细阅读错误信息，定位两个定义位置**
- **启动失败时关注第一个抛出的异常，后续可能是连锁反应**

## 问题13：座位布局保存500 - 会场唯一约束缺少日程维度

**问题表现：**
- 点击保存布局返回 `500 Internal Server Error`
- 后端报错：`Duplicate entry '2030309010523144194-观众区' for key 'uk_tenant_conf'`

**问题原因：**
- `conf_seating_venue` 表原有唯一约束 `uk_tenant_conf(conference_id, venue_name)`
- 多日程场景下，同一会议的不同日程可能使用同名会场（如"观众区"）
- 第二个日程保存时触发唯一键冲突

**修复方案：**
1. 删除旧约束：`DROP INDEX uk_tenant_conf ON conf_seating_venue`
2. 给 `conf_seating_venue` 表新增 `schedule_id` 列
3. 创建新唯一约束：`uk_conf_sched_venue(conference_id, schedule_id, venue_name)`
4. 重写 `saveLayout()` 和 `loadLayout()` 方法支持按日程隔离

**数据库变更：**
```sql
ALTER TABLE conf_seating_venue ADD COLUMN schedule_id BIGINT AFTER conference_id;
ALTER TABLE conf_seating_venue ADD UNIQUE KEY uk_conf_sched_venue(conference_id, schedule_id, venue_name);
```

**文件位置：**
- `conference-seating/.../entity/SeatingVenue.java`
- `conference-seating/.../mapper/SeatingVenueMapper.java`
- `conference-seating/.../service/impl/SeatingSeatServiceImpl.java`

**经验教训：**
- **设计数据库唯一约束时必须考虑所有业务维度**（会议×日程×名称）
- 多日程场景是设计阶段就应考虑的核心需求，不应后补

## 问题14：座位布局保存500 - 座位唯一约束维度错误

**问题表现：**
- 保存座位时报错：`Duplicate entry 'xxx-1' for key 'uk_seat_number'`
- 一个区域中多行的第1号座位冲突

**问题原因：**
- 原唯一约束为 `uk_seat_number(venue_id, seat_number)`
- 前端 `seatNumber` 生成逻辑使用 `seat.number`（即列号），不同行的同列座位编号相同
- 例如1排1号和2排1号的 `seat_number` 都是 `"1"`，在同一 `venue_id` 下冲突

**修复方案：**
1. 删除旧约束：`DROP INDEX uk_seat_number ON conf_seating_seat`
2. 创建正确约束：`ADD UNIQUE KEY uk_venue_row_col(venue_id, row_num, column_num)`
3. 前端 `seatNumber` 改为 `${rowIdx+1}-${colIdx+1}` 格式（如 `"2-3"` 表示2排3号）

**数据库变更：**
```sql
ALTER TABLE conf_seating_seat DROP INDEX uk_seat_number;
ALTER TABLE conf_seating_seat ADD UNIQUE KEY uk_venue_row_col(venue_id, row_num, column_num);
```

**文件位置：**
- `admin-pc/.../pages/seating-mgr.html`（`saveLayoutToBackend` 函数）

**经验教训：**
- **唯一约束的组合字段必须能真正唯一标识一条记录**
- 座位的唯一性由"哪个会场+第几排+第几列"决定，不是"哪个会场+座位编号"
- 前端生成的编号格式必须与后端约束逻辑匹配

## 问题17：智能排座只分配1人 - JavaScript雪花ID精度丢失（核心Bug）

**问题表现：**
- 报名系统有40条已通过记录，但智能排座后只排了1个人
- 控制台显示 `attendees.value.length = 40` 但排座结果只有1人

**问题原因（根因链路）：**
1. 报名表使用 MyBatis-Plus 雪花算法生成ID，如 `2031308955921170434`（19位数字）
2. JavaScript `Number.MAX_SAFE_INTEGER = 2^53 - 1 = 9007199254740991`（16位）
3. 雪花ID超过安全整数范围，`JSON.parse()` 时精度丢失
4. 所有ID如 `2031308955921170434`、`2031308955921170435` 等，解析后都变成 `2031308955921170400`
5. 前端去重逻辑使用 `r.id` 作为去重 key → 40人的key全部相同 → 只保留1人

**问题链路图：**
```
后端返回JSON → {"id": 2031308955921170434, ...}
                              ↓
JavaScript JSON.parse() → id = 2031308955921170400  (精度丢失！)
                              ↓
所有40条记录的id → 全部变成 2031308955921170400
                              ↓
去重 Set.add(r.id) → Set 只有1个元素 → 结果只剩1人
```

**修复方案：**
- ~~将去重 key 从 `r.id` 改为 `(r.realName || r.name) + '_' + r.phone`~~
- 最终方案：从源头清理重复数据（见问题18），彻底移除去重逻辑

**经验教训（极其重要）：**
- **雪花ID（Snowflake）超过 `2^53-1` 时，JavaScript 无法精确表示**
- **后端返回大整数ID时，必须以字符串格式传输**（`"id": "2031308955921170434"`）
- 可在 Jackson 全局配置：`@JsonSerialize(using = ToStringSerializer.class)` 或 `objectMapper.configure(SerializationFeature.WRITE_NUMBERS_AS_STRINGS)`
- **前端永远不要用数值型ID做精确比较或去重 key**
- 这是 Java+JS 前后端分离项目的经典陷阱，建表阶段就应决定ID传输方案

## 问题18：报名数据源头重复 - 导入脚本被执行两次

**问题表现：**
- 前端从报名服务获取到40条已通过记录
- 用户质疑："报名系统明明只有20个人，为什么变成了40个？"

**问题原因：**
- `insert_registrations_with_node.js` 脚本定义了**20个人**的模拟数据
- 该脚本被**执行了两次**（时间戳 `11:22:56` 和 `11:23:49`，间隔53秒）
- `conf_registration` 表**没有唯一约束**防止同一会议中同一手机号重复注册
- 结果：20人 × 2次 = 40条记录，每人有2条不同ID的记录

**验证查询：**
```sql
SELECT name, phone, COUNT(*) as cnt 
FROM conf_registration 
WHERE status=1 AND deleted=0 
GROUP BY name, phone HAVING cnt > 1;
-- 结果：全部20人各出现2次
```

**修复方案：**
1. **清理源头数据**：`DELETE` 每人重复的第二条记录（保留ID较小的）
2. **添加唯一约束**：`ALTER TABLE conf_registration ADD UNIQUE KEY uk_meeting_phone(meeting_id, phone)`
3. **修复导入脚本**：执行前先 `DELETE` 旧数据，幂等化
4. **移除前端去重代码**：数据源正确后，去重逻辑不再需要

**文件位置：**
- `insert_registrations_with_node.js`
- `admin-pc/.../pages/seating-mgr.html`（两处去重代码已删除）

**经验教训：**
- **测试数据脚本必须幂等**（可重复执行不产生副作用）
- 方式1：执行前先 `DELETE WHERE meeting_id = ?`
- 方式2：使用 `INSERT ... ON DUPLICATE KEY UPDATE`
- **关键业务表必须有唯一约束**（如 `meeting_id + phone` 组合唯一）
- **出现数据异常时，首先检查数据源是否正确，不要在消费端打补丁**

## 问题19：页面加载时从数据库查不到已保存的布局

**问题表现：**
- 日志显示 `[loadSeatingData] 数据库无座位布局，尝试localStorage`
- 但数据库中明确存有 `schedule_id=1001` 的座位布局数据

**问题原因（初始化顺序错误）：**
1. `onMounted` 中 `initializeConference()` 内部直接调用 `loadSeatingData()`
2. 此时 `loadSchedules()` 还未执行，`currentScheduleId` 仍为 `null`
3. API 请求变成 `/layout/load?conferenceId=XXX`（无 scheduleId 参数）
4. 后端 SQL 条件变成 `AND schedule_id IS NULL`，而数据存在 `schedule_id=1001` 下
5. 返回空数组 → 回退到 localStorage

**错误的执行顺序：**
```
onMounted
  → initializeConference()
      → loadSeatingData()          ← currentScheduleId = null ❌
  → loadSchedules()                ← 这里才设置 currentScheduleId = 1001
```

**修复方案：**
```
onMounted
  → initializeConference()         ← 只加载会议信息，不加载座位
  → loadSchedules()                ← 设置 currentScheduleId = 1001
  → loadSeatingData()              ← 此时有正确的 scheduleId ✅
```
- 从 `initializeConference()` 中移除三处 `loadSeatingData()` 调用
- 在 `onMounted` 中按正确顺序：先日程 → 再布局

**文件位置：**
- `admin-pc/.../pages/seating-mgr.html`（`initializeConference` + `onMounted`）
- `admin-pc/.../modules/seating-schedule-manager.js`（`loadSchedules` 首次加载时不提前加载座位）

**经验教训：**
- **异步初始化必须严格控制执行顺序**
- 有依赖关系的初始化步骤必须串行 `await`，不能各自独立调用
- 命名应体现职责边界：`initializeConference` 不应包含座位加载逻辑

## 问题20：TenantFilter 只从JWT读取租户ID，忽略Header

**问题表现：**
- API `/layout/load` 传了 `X-Tenant-Id` header 但后端 `tenantId` 为 null
- SQL 查询 `tenant_id = null` 匹配不到任何记录
- 与问题19叠加，即使 scheduleId 正确也查不到数据

**问题原因：**
- `TenantFilter` 原实现只从 JWT `Authorization` header 提取 tenantId
- 前端排座页面没有 JWT token（静态页面直接调用API）
- 没有 JWT 时，`TenantContextHolder.getTenantId()` 返回 `null`

**修复方案：**
在 `TenantFilter` 中增加回退逻辑：
```java
// 如果JWT未设置tenantId，尝试从 X-Tenant-Id header 读取
if (TenantContextHolder.getTenantId() == null) {
    String headerTenantId = request.getHeader("X-Tenant-Id");
    if (headerTenantId != null) {
        TenantContextHolder.setTenantId(Long.parseLong(headerTenantId));
    }
}
```

**文件位置：**
- `conference-common/.../tenant/TenantFilter.java`

**经验教训：**
- **多租户过滤器应支持多种租户ID来源**（JWT、Header、参数）并有明确的优先级
- 不同前端场景（SPA有JWT vs 静态页面用Header）需要统一的后端兼容方案
- 增加日志输出实际使用的tenantId来源，方便排查

## 问题21：localStorage脏数据污染参会人员列表

**问题表现：**
- 数据库布局加载失败后，回退到 localStorage
- 但 localStorage 中保存了旧数据（含1个参会人员）
- 导致 `attendees.value.length = 1`，跳过从报名服务加载的逻辑

**问题原因：**
- `loadSeatingDataFromLocalStorage()` 同时加载了 `seatingAreas` 和 `attendees`
- 加载后 `attendees.value.length !== 0`
- 后续判断 `if (attendees.value.length === 0)` 为 false，跳过报名服务 API 调用
- 参会人员被 localStorage 中的旧脏数据覆盖

**修复方案：**
1. `loadSeatingDataFromLocalStorage()` **不再加载** `attendees`（只加载座位区布局）
2. `loadSeatingData()` **始终**从报名服务获取参会人员，移除 `attendees.value.length === 0` 前置条件
3. 参会人员的唯一数据源是报名服务 API，不从 localStorage 读取

**文件位置：**
- `admin-pc/.../pages/seating-mgr.html`（`loadSeatingDataFromLocalStorage` + `loadSeatingData`）

**经验教训：**
- **localStorage 不应作为业务数据的权威来源**，只能做缓存/暂存
- **不同类型的数据（布局 vs 人员）应有独立的加载策略**，不能捆绑
- **"有数据就不加载"的逻辑有隐患**——如果已有的是脏数据就永远得不到正确数据
- 正确做法：先从权威源加载，失败时才回退到缓存，且标记为"缓存数据"

## 智能排座功能 - 经验总结

### 1. 数据库设计原则
- **唯一约束必须覆盖所有业务维度**（会议×日程×名称、会场×行×列）
- **可选关联字段必须允许 NULL**
- **跨微服务的ID引用不建立数据库外键**
- **关键业务表必须有防重复约束**
- **测试数据脚本必须幂等化**

### 2. 雪花ID与前端交互
- **Java雪花ID超过 `2^53-1`，JavaScript 无法精确表示**
- 后端返回大整数ID必须序列化为字符串
- 前端不能用数值型ID做精确比较、去重、Map key
- 推荐全局配置：`@JsonSerialize(using = ToStringSerializer.class)` 标注 Long 类型 ID 字段

### 3. 异步初始化顺序
- 有依赖关系的初始化必须串行 `await`
- 函数命名应体现职责边界（`initConference` 不应包含 `loadSeating`）
- `onMounted` 中的调用顺序就是业务流程顺序

### 4. 数据源权威性
- 每类数据只有一个权威数据源（参会人员 → 报名服务API）
- localStorage 只做缓存，不做权威源
- 数据异常时先查源头，不在消费端打补丁（去重）

### 5. 多租户架构
- TenantFilter 应支持多种租户ID来源（JWT > Header > 参数）
- 不同前端场景（SPA / 静态页面）的认证方式不同，后端必须兼容
- 过滤器中的异常会被前端误判为CORS问题

### 6. 问题排查思路
- **500错误**：看后端日志第一个异常，不要只看前端报错
- **数据不对**：先查数据库源头，确认数据本身是否正确
- **查不到数据**：检查查询条件每一项（tenantId? scheduleId? status?）
- **加载顺序**：在关键节点打 `console.log` 确认执行时序

## 问题22：PowerShell编辑UTF-8文件导致全部中文乱码（严重事故）

> 记录日期：2026年3月16日
> 功能模块：智能分组功能 - 报名汇总页面
> 涉及文件：`admin-pc/conference-admin-pc/pages/registration-mgr.html`

**问题表现：**
- 浏览器控制台报错：`Uncaught SyntaxError: Unexpected identifier 'fas'`（第768行）
- 页面完全无法加载，白屏
- 用编辑器打开文件发现所有中文变成乱码（如 `缃戦〉鎶ュ悕` 应为 `网页报名`）

**问题原因：**
- 使用 PowerShell 5.1 的 `(Get-Content file) -replace ... | Set-Content -Encoding UTF8` 命令批量替换文件中的 API URL
- **PowerShell 5.1 的 `Get-Content` 默认使用系统 ANSI 编码（Windows 中文环境为 GBK/CP936）读取文件**
- 实际文件为 UTF-8 编码，被当作 GBK 读入后再以 UTF-8 BOM 写出，造成**不可逆的双重编码损坏**
- 文件中所有 UTF-8 编码的中文字符（约4000+处）被永久破坏

**错误的命令（禁止使用）：**
```powershell
# ❌ 以下命令会损坏含中文的 UTF-8 文件
(Get-Content "file.html") -replace "old", "new" | Set-Content "file.html" -Encoding UTF8
```

**尝试的恢复措施及结果：**
1. **编码反转法**（UTF8→GBK→UTF8）：使用 `[Text.Encoding]::GetEncoding(936)` 尝试逆向转换 → **失败**，坏字符从354个恶化到4313个，因为 GBK→UTF8→GBK 不是无损可逆操作
2. **从干净备份恢复**：从 `NewZip_extract/` 目录复制原始干净文件覆盖 → **成功**，但之前所有修改（约200行新增/修改代码）全部丢失，需要重新手工应用

**最终修复方案：**
1. 从 `NewZip_extract/admin-pc/conference-admin-pc/pages/registration-mgr.html` 恢复干净的原始文件（2848行，UTF-8编码完好）
2. 使用 VS Code 编辑工具（`replace_string_in_file` / `multi_replace_string_in_file`）重新应用所有9处 API 集成修改
3. 验证恢复后文件：3037行，0个坏字符，中文编码正常

**损失评估：**
- 约2小时工作量损失（分析乱码原因 + 差异对比 + 重新应用修改）
- 干净备份的存在是唯一的救命手段

**根本教训与强制规则：**

| 规则 | 说明 |
|------|------|
| **🚫 永远不要用 PowerShell `Get-Content` / `Set-Content` 编辑含中文的 UTF-8 文件** | PowerShell 5.1 默认编码为系统 ANSI（GBK），不是 UTF-8。即使指定 `-Encoding UTF8` 也只影响写入，读取仍用 ANSI |
| **✅ 编辑文件必须使用 VS Code 工具** | `replace_string_in_file`、`multi_replace_string_in_file` 等工具能正确处理 UTF-8 编码 |
| **✅ 修改前保留干净备份** | 关键文件修改前先复制一份备份，且备份路径不要与工作路径混淆 |
| **✅ PowerShell 如需读写 UTF-8 文件，必须用 .NET API** | 正确方式：`[System.IO.File]::ReadAllText(path, [System.Text.Encoding]::UTF8)` 和 `[System.IO.File]::WriteAllText(path, content, [System.Text.Encoding]::UTF8)` |
| **✅ 每次修改后立即验证编码** | 检查坏字符数：`($content.ToCharArray() \| Where-Object { $_ -eq [char]0xFFFD }).Count` 必须为 0 |

**文件位置：** `admin-pc/conference-admin-pc/pages/registration-mgr.html`

**影响范围：** 报名汇总页面完全不可用（白屏），智能分组功能所有前端修改丢失

**严重程度：** ⚠️ **严重** — 造成数据不可逆损坏，仅因存在原始备份才得以恢复
