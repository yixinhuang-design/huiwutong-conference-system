## 🐛 Bug 追踪和解决

### 发现 Bug 时的流程

1. **记录信息**
   - 使用浏览器开发者工具截图
   - 记录重现步骤
   - 保存 Console 错误信息

2. **分类**
   - **严重 (Critical)**: 功能完全无法使用
   - **高 (High)**: 主要功能受损
   - **中 (Medium)**: 部分功能异常
   - **低 (Low)**: 界面或体验问题

3. **解决**
   - 后端 bug：修改 Java 代码，重新编译
   - 前端 bug：修改 HTML/JS，刷新浏览器
   - 数据库 bug：修改 SQL，重新执行迁移

4. **验证**
   - 修复后重新测试
   - 检查是否引入新的问题
   - 记录解决方案

### 常见问题速度解决表

| Bug | 原因 | 解决方案 | 预计修复时间 |
|-----|------|--------|-----------|
| 页面显示模拟数据 | API 加载失败 | 检查后端日志，验证端口和权限 | 15 min |
| 创建后没有出现 | 响应格式错误 | 检查 API 返回值格式 | 20 min |
| 按钮点击无反应 | JS 错误 | 检查 Console 错误 | 30 min |
| 数据不同步 | 本地更新失败 | 检查 API 响应的数据结构 | 30 min |
| 样式错乱 | CSS 加载失败 | 清除浏览器缓存，重新加载 | 5 min |

## 🚀 部署步骤

### 步骤 1: 生产环境准备（Day 1 下午）
```bash
# 1. 备份现有数据
mysqldump -h prod_host -P 3308 -u root conference_registration > backup_before_deploy.sql

# 2. 检查 Java 版本
java -version  # 应该是 21+

# 3. 检查 MySQL 连接
mysql -h prod_host -P 3308 -u root -p -e "SELECT VERSION();"
```

### 步骤 2: 数据库迁移（Day 1 下午）
```bash
# 1. 连接到生产数据库
mysql -h prod_host -P 3308 -u root -pPASSWORD conference_registration

# 2. 执行迁移脚本
source V7__schedule_schema.sql;

# 3. 验证表创建
SHOW TABLES LIKE 'conf_schedule%';
# 应该显示 8 个表

# 4. 验证数据
SELECT COUNT(*) FROM conf_schedule;
# 应该返回 0（初始为空）
```

### 步骤 3: 后端部署（Day 2 早上）
```bash
# 1. 编译
cd conference-meeting
mvn clean compile -Pproduction

# 2. 打包
mvn package -DskipTests

# 3. 部署
cp target/conference-meeting-0.0.1-SNAPSHOT.jar /opt/app/

# 4. 启动（使用生产配置）
java -jar /opt/app/conference-meeting-0.0.1-SNAPSHOT.jar \
  --spring.profiles.active=production \
  --spring.datasource.url=jdbc:mysql://prod_host:3308/conference_registration
```

### 步骤 4: 前端部署（Day 2 早上）
```bash
# 1. 复制文件到 Web 服务器
cp -r admin-pc/conference-admin-pc/* /var/www/html/admin-pc/
cp -r app-uniapp /var/www/app-source/

# 2. 验证文件权限
chmod -R 755 /var/www/html/admin-pc/
chmod -R 755 /var/www/app-source/

# 3. 清除 CDN 缓存（如有）
# 联系 CDN 运维清除缓存
```
