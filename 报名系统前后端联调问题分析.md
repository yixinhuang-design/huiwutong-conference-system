# 报名系统前后端联调问题完整分析

## 执行摘要

通过全面分析项目代码，发现报名系统前后端存在**多个字段映射不一致和验证逻辑错位**的问题，导致联调困难。本文档列出所有问题、根本原因和具体修复方案。

---

## 问题清单

### 问题1：数据库表与后端实体字段映射不一致

**问题位置**：
- 数据库表：`conference_registration.conf_registration` (V2__registration_schema.sql第38-120行)
- 后端实体：`Registration.java` (第1-146行)

**具体表现**：

| 数据库列 | 后端Java字段 | 映射注解 | 问题 |
|---|---|---|---|
| `name` | `realName` | `@TableField("name")` | ✗ 列名与字段名不符 |
| `organization` | `department` | `@TableField("organization")` | ✗ 列名与字段名不符 |
| `extra_info` | `registrationData` | `@TableField("extra_info")` | ✗ 列名与字段名不符 |

**根本原因**：
后端实体设计时使用了更语义化的Java字段名，但未与数据库表设计保持命名一致。MyBatis-Plus的`@TableField`注解虽然能处理映射，但这种不一致容易造成理解混乱。

**影响范围**：
- 数据库查询时字段映射正确（因为有@TableField注解）
- 前后端通信时可能产生字段命名混淆
- 代码可读性降低

**数据流证明**：
```sql
-- 数据库表实际存储
CREATE TABLE `conf_registration` (
  `name` VARCHAR(50) NOT NULL,           -- 存储姓名
  `organization` VARCHAR(200),           -- 存储单位/部门
  `extra_info` JSON                      -- 存储额外信息
)

-- Java实体映射
@TableField("name") private String realName;
@TableField("organization") private String department;
@TableField("extra_info") private String registrationData;
```

### 问题2：前端发送与后端期望的默认字段不一致

**问题位置**：
- 前端：`scan-register.vue` 第79行、第155-164行、第167-180行
- 后端：`RegistrationServiceImpl.java` 第558-565行（defaultFormFields方法）

**具体表现**：

后端的defaultFormFields返回必填字段：
```java
defaults.add(systemField("name", "姓名", "text", true, 1));
defaults.add(systemField("phone", "手机号", "text", true, 2));
defaults.add(systemField("unit", "单位", "text", true, 3));
defaults.add(systemField("position", "职位", "text", true, 4));
```

前端的固定字段过滤集合：
```javascript
const fixed = new Set(['name', 'realname', 'phone', 'unit', 'department', 'position', 'remark', 'email']);
```

**问题**：
1. 后端期望registrationData中包含"name"字段（必填）
2. 前端虽然在registrationData中包含了"name"字段（第159行）
3. 但后端的validateDynamicFields无法正确验证，因为字段名处理方式不一致

**根本原因**：
- 后端的defaultFormFields方法独立维护一份字段列表，与前端的字段定义不同步
- 前端过滤了这些字段认为是"固定"的，不作为动态字段处理
- 后端仍然会在validateDynamicFields中验证这些字段

**数据流证明**：

前端submit方法中发送的registrationData（第167-180行）：
```javascript
const registrationData = {
  realName: this.form.realName,
  name: this.form.realName,              // ✓ 包含name字段
  phone: this.form.phone,
  unit: this.form.department,
  department: this.form.department,
  position: this.form.position,
  remark: this.form.remark,
  ...this.dynamicValues,
  files: { idCardPhotoUrl: this.upload.fileUrl || '' },
  ocr: this.upload.ocr || null
};

await request({
  data: {
    // ... 其他字段
    registrationData: JSON.stringify(registrationData)  // JSON字符串格式
  }
});
```

后端validateDynamicFields方法（第532-556行）：
```java
private void validateDynamicFields(String registrationData, List<RegistrationFieldConfigRequest.FieldConfig> fields) {
    // fields来自defaultFormFields()，包含必填的"name"
    JSONObject data = safeObject(registrationData);
    for (RegistrationFieldConfigRequest.FieldConfig field : fields) {
        String key = field.getKey();
        String value = data.getString(key);
        if (Boolean.TRUE.equals(field.getRequired()) && !StringUtils.hasText(value)) {
            throw new BusinessException(ResultCode.PARAM_ERROR, "动态字段必填: " + field.getLabel());
        }
    }
}
```

### 问题3：后端服务未从数据库加载字段配置

**问题位置**：
- 后端：`RegistrationServiceImpl.java` 第349-380行 (getRegistrationFormFields方法)
- SQL查询：`registrationMapper.getMeetingRegistrationConfig(conferenceId, tenantId)`

**具体表现**：
```java
public List<RegistrationFieldConfigRequest.FieldConfig> getRegistrationFormFields(Long conferenceId) {
    String configText = registrationMapper.getMeetingRegistrationConfig(conferenceId, tenantId);
    if (!StringUtils.hasText(configText)) {
        return defaultFormFields();  // ← 如果查询无结果，使用硬编码默认值
    }
    // ... 解析configText
}
```

**问题**：
- 如果数据库中没有为该会议配置字段，就回退到硬编码的defaultFormFields
- 没有日志记录是否成功加载了数据库配置
- 前后端无法一致地同步字段定义

**根本原因**：
- 系统中可能没有创建会议对应的字段配置
- 或者字段配置表的查询SQL有问题
- 导致总是使用默认的硬编码字段列表

**影响**：
每次前端请求动态字段时，后端都会返回固定的默认字段列表，即使前端期望的字段配置不同。

### 问题4：idCard字段的required属性不一致

**问题位置**：
- 前端：`scan-register.vue` 第218-223行（submit方法只验证必填的realName、phone、department、position）
- 后端：`RegistrationRequest.java` 第47行（`@NotBlank(message = "身份证号不能为空")`）

**具体表现**：

前端submit方法中的验证（第218-223行）：
```javascript
if (!this.form.realName || !this.form.phone || !this.form.department || !this.form.position) {
  uni.showToast({ title: '请填写必填项', icon: 'none' });
  return;
}
// 没有验证 idCard 字段是否为空
```

发送的数据（第175行）：
```javascript
idCard: this.upload?.ocr?.idCard || '未识别',  // 如果OCR失败，发送'未识别'
```

后端RegistrationRequest的验证注解：
```java
@NotBlank(message = "身份证号不能为空")
private String idCard;
```

**问题**：
1. 前端没有验证idCard，可能发送'未识别'这样的值
2. 后端@NotBlank验证会失败
3. 导致提交失败，但错误信息与用户预期不符

**根本原因**：
- 前端认为身份证识别失败时，可以发送'未识别'字符串
- 后端要求必须是真实的身份证号
- 两端对身份证字段的处理逻辑不一致

**数据流证明**：
```
用户上传身份证但OCR失败
  ↓
this.upload.ocr = null
  ↓
idCard: this.upload?.ocr?.idCard || '未识别'  (前端)
  ↓
POST /api/registration/submit { idCard: '未识别' }
  ↓
@NotBlank验证
  ↓
"身份证号不能为空" 错误（虽然字段不为空）
```

### 问题5：前端OCR结果处理与后端验证不同步

**问题位置**：
- 前端：`scan-register.vue` 第129-142行（doUpload方法）
- 后端：`RegistrationServiceImpl.java` 第115-128行（submitRegistration方法）

**具体表现**：

前端上传后立即验证（第131-139行）：
```javascript
const ocr = await request({
  service: 'registration',
  url: `${API_CONFIG.ENDPOINTS.REGISTRATION_OCR}?photoUrl=${encodeURIComponent(this.upload.fileUrl)}`,
  method: 'POST'
});
this.upload.ocr = ocr;
```

后端重新验证OCR（第115-128行）：
```java
if (StringUtils.hasText(request.getIdCardPhotoUrl())) {
    Map<String, String> ocrResult = ocrIdCard(request.getIdCardPhotoUrl());
    LinkedHashMap<String, Object> validation = validateOcrResult(registration, ocrResult);
    // ... 合并验证结果到registrationData
}
```

**问题**：
1. 前端已经调用过OCR识别，并获得了结果
2. 但后端在submitRegistration中**再次调用**ocrIdCard进行识别
3. 这样做会导致：
   - 重复调用OCR服务，浪费资源
   - 可能得到不同的结果（如果识别服务不幂等）
   - 后端验证结果与前端显示给用户的结果不一致

**根本原因**：
- 前端认为可以信任用户上传的图片已经识别过了
- 后端不相信前端的识别结果，坚持自己验证
- 两端对信息流的理解不一致

### 问题6：registrationData序列化与后端解析的格式不一致

**问题位置**：
- 前端：`scan-register.vue` 第180行（发送JSON.stringify序列化的对象）
- 后端：`RegistrationServiceImpl.java` 第98行（使用safeObject解析）

**具体表现**：

前端发送（第180行）：
```javascript
registrationData: JSON.stringify(registrationData)  // 字符串格式
```

后端接收（第98行）：
```java
validateDynamicFields(request.getRegistrationData(), formFields);

private void validateDynamicFields(String registrationData, ...) {
    JSONObject data = safeObject(registrationData);
    // ...
}

private JSONObject safeObject(String json) {
    if (!StringUtils.hasText(json)) {
        return new JSONObject();
    }
    try {
        return JSON.parseObject(json);
    } catch (Exception e) {
        return new JSONObject();
    }
}
```

**问题**：
1. safeObject方法在JSON解析失败时**悄悄返回空对象**，不记录错误
2. 这会导致验证通过，但数据实际上是空的
3. 前端用户无法看到数据解析失败的错误

**根本原因**：
- 错误处理过于宽松
- 没有日志记录解析失败的情况
- 无法诊断数据格式问题

### 修复1：规范后端实体的字段命名（推荐）

**影响分析**：

- ✗ 短期改动范围大

**影响范围**：
- RegistrationServiceImpl中的所有字段赋值处

### 修复2：统一后端的字段定义和验证逻辑

**文件**：`RegistrationServiceImpl.java`

**当前问题**：
defaultFormFields方法返回硬编码的字段列表，与前端的字段定义不同步。

**修改方案**：

```java
public List<RegistrationFieldConfigRequest.FieldConfig> getRegistrationFormFields(Long conferenceId) {
    Long tenantId = currentTenantId();
    String configText = registrationMapper.getMeetingRegistrationConfig(conferenceId, tenantId);

    if (!StringUtils.hasText(configText)) {
        logger.warn("会议[{}]未配置动态字段，使用系统默认字段", conferenceId);
        return defaultFormFields();
    }

    JSONObject config = safeObject(configText);
    JSONArray fields = config.getJSONArray("formFields");
    if (fields == null || fields.isEmpty()) {
        logger.warn("会议[{}]字段配置为空，使用系统默认字段", conferenceId);
        return defaultFormFields();
    }

    // ... 后续解析逻辑
    return list;
}

// 修改defaultFormFields - 仅包含真正的系统必填字段
private List<RegistrationFieldConfigRequest.FieldConfig> defaultFormFields() {
    List<RegistrationFieldConfigRequest.FieldConfig> defaults = new ArrayList<>();
    // 注意：只包括最基础的必填字段，不包括在RegistrationRequest中已有的字段
    // 因为那些字段已经由@NotBlank等注解验证了
    return defaults;  // 返回空列表，让前端自己管理所有字段
}
```

**原因**：
- 后端RegistrationRequest中已经用@NotBlank注解验证了基本字段
- 不需要再在validateDynamicFields中重复验证
- defaultFormFields应该只用于动态字段，不包括系统基础字段

### 修复4：后端OCR验证去重

**文件**：`RegistrationServiceImpl.java`（第115-128行）

**当前问题**：
前端已经调用过OCR识别，后端还要再次识别。

**修改方案**：

```java
@Transactional
public Registration submitRegistration(RegistrationRequest request) {
    // ... 前面的代码

    String mergedData = mergeRegistrationDataWithFiles(
        request.getRegistrationData(), 
        request.getIdCardPhotoUrl(), 
        request.getDiplomaPhotoUrl()
    );

    // 只在前端没有提供OCR结果时才进行OCR识别
    if (StringUtils.hasText(request.getIdCardPhotoUrl()) && !hasOcrResult(mergedData)) {
        Map<String, String> ocrResult = ocrIdCard(request.getIdCardPhotoUrl());
        LinkedHashMap<String, Object> validation = validateOcrResult(registration, ocrResult);

        JSONObject dataJson = safeObject(mergedData);
        dataJson.put("ocr", ocrResult);
        dataJson.put("ocrValidation", validation);
        dataJson.put("ocrTime", LocalDateTime.now().toString());
        mergedData = dataJson.toJSONString();
    }

    registration.setRegistrationData(mergedData);
    registrationMapper.insert(registration);
    return registration;
}

private boolean hasOcrResult(String registrationData) {
    JSONObject data = safeObject(registrationData);
    JSONObject ocr = data.getJSONObject("ocr");
    return ocr != null && !ocr.isEmpty();
}
```

```javascript
      method: 'GET'
    });
    const fixed = new Set(REGISTRATION_FIXED_FIELDS.map(f => f.toLowerCase()));
    this.dynamicFields = (Array.isArray(data) ? data : [])
      .filter(f => f && f.enabled !== false && !fixed.has((f.key || '').toLowerCase()))
      .sort((a, b) => (a.sort || 0) - (b.sort || 0));
  } catch (e) {
    uni.showToast({ title: `动态字段加载失败: ${e.message}`, icon: 'none' });
  }
}
```

## 总结

报名系统的前后端联调问题主要源于：

1. **字段命名不一致** - 数据库、后端实体、前端请求的字段名互不对应
2. **字段定义不同步** - 后端硬编码字段列表与前端的理解不一致  
3. **信息流重叠** - 前端认为已处理（OCR识别），后端又重新处理
4. **错误处理宽松** - 问题被默默忽略，难以诊断

解决这些问题需要：
- 统一命名规范
- 前后端字段定义同步
- 明确的职责边界
- 完善的日志和错误处理

按照本文档的修复方案，可以逐步改善系统的稳定性和可维护性。
